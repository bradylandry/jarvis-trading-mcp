## Medium

### `get_fit_score` implementation calls wrong endpoint
- **File**: `friend_mcp/server.py:175`
- **Issue**: The tool is described as returning a "Directional fit score (0-10) with per-component evidence (sentiment, fundamentals, Kimi earnings, Kimi-vs-analyst delta, insider activity)" but its implementation calls `_api_get("/api/screener", {"tickers": ticker.upper()})`. The in-code comment acknowledges this is a workaround because no direct fit-score endpoint exists. The screener endpoint likely returns a different response shape (stock screener results) rather than per-component directional evidence. If the screener response doesn't include those components, Claude will receive data that doesn't match the tool's stated contract — leading to confusing tool results.
- **Fix**: Either (a) update the tool description to accurately describe what `/api/screener` returns for a single ticker, or (b) add a server-side `/api/fit-score?ticker=X` endpoint so the implementation matches the description. At minimum, add a note in the docstring/description that this calls the screener endpoint and explain what fields to look for in the response.

## Low

### No client-side validation of `expiration` format in `get_thesis`
- **File**: `friend_mcp/server.py:163`
- **Issue**: The `expiration` parameter is documented as "YYYY-MM-DD" but no format check is performed. An obviously wrong value (e.g. `"tomorrow"`, a 200-char string) gets silently passed to the server. The schema marks it as a `string` with no `pattern` constraint, so Claude won't reject it either. A structured-error response from the API is returned, which is fine — but a simple regex check or `datetime.strptime` guard here would surface the problem before the network hop.
- **Fix**: Validate format with a lightweight check such as `if not re.match(r"^\d{4}-\d{2}-\d{2}$", expiration): return {"error": "expiration must be YYYY-MM-DD"}`. Also add `"pattern": "^\\d{4}-\\d{2}-\\d{2}$"` to the JSON schema `inputSchema`.

### No lower/upper bounds on `dte` in `get_iv`
- **File**: `friend_mcp/server.py:219`
- **Issue**: `dte` is cast to `int(dte)` but not validated for range. Negative values or absurdly large values (e.g., `dte=99999`) are forwarded to the server. The server presumably handles this, but a client-side guard would give a faster, cleaner error to Claude.
- **Fix**: Add `dte = max(1, min(int(dte), 365))` or return an error for out-of-range values. Update the schema's `description` to say "1–365".
