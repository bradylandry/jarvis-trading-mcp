## High

### `get_fit_score` ignores `opt_type` — always returns undirected screener data
- **File**: `friend_mcp/server.py:183`
- **Issue**: The `get_fit_score(ticker, opt_type)` function accepts an `opt_type` parameter (call/put) described to Claude as driving "directional fit" scoring, but the implementation passes only `tickers=ticker.upper()` to `/api/screener` — `opt_type` is never forwarded. Every call returns the same payload regardless of direction. Claude will present put-direction results using the same data as call-direction results, potentially inverting the directional signal for the user.
- **Fix**: Pass `opt_type` (or equivalent) as a query parameter: `_api_get("/api/screener", {"tickers": ticker.upper(), "type": opt_type})`. If the `/api/screener` endpoint does not accept a direction parameter, update the tool description and schema to drop `opt_type`, and document that the score is direction-agnostic.

### `urljoin(API_BASE, path)` silently drops path segments if `API_BASE` contains a path
- **File**: `friend_mcp/server.py:128`
- **Issue**: `urllib.parse.urljoin(base, url)` follows RFC 3986 semantics: if `base` has a non-empty path (e.g. `https://trading.landrycmd.com/v2`), the relative `path` (e.g. `/api/options/scan/universe`) replaces only the last segment of the base path, not the full path. For the default value `https://trading.landrycmd.com` (no path) this is benign. But if a user sets `JARVIS_TRADING_API=https://proxy.example.com/jarvis`, all calls silently resolve to `https://proxy.example.com/api/...` (dropping `/jarvis`), hitting the wrong endpoints with no error. The failure mode is silent misdirection rather than an obvious crash.
- **Fix**: Use explicit path joining that is immune to this: `f"{API_BASE.rstrip('/')}{path}"` (the pattern replaced in 0.2.1 for different reasons), or validate `API_BASE` at startup to assert it has no path component: `assert urllib.parse.urlparse(API_BASE).path in ('', '/'), "JARVIS_TRADING_API must not contain a path"`. Document the env var contract.

### No client-side input validation — user-supplied args passed raw to the API
- **File**: `friend_mcp/server.py:352-353`
- **Issue**: `fn(**args)` passes Claude-supplied arguments directly to tool functions with no type coercion or validation beyond what Python's function call machinery provides. For example: `get_thesis` requires `expiration` in `YYYY-MM-DD` format — a malformed string produces an opaque upstream error. `get_iv`'s `dte` parameter is `int(dte)` inside the function body (line 219), but if `dte` arrives as a non-numeric string, the `int()` call raises `ValueError`, which is caught by the outer `except Exception` in `_handle` and returned as a generic tool failure with no hint to Claude about what went wrong. Enum-typed fields (`mode`, `opt_type`) are not validated against their allowed values before the request is sent.
- **Fix**: Add a thin validation layer before `fn(**args)`: check required fields are present and non-empty, coerce/validate types, and reject known-bad enum values with a structured `{"error": "..."}` response that gives Claude actionable feedback. A `pydantic` model per tool or a simple manual check dict is sufficient given the small surface area.

## Medium

### Version mismatch: `__init__.py` advertises `0.1.0`, `pyproject.toml` and `serverInfo` diverge
- **File**: `friend_mcp/__init__.py:15`, `pyproject.toml:7`, `friend_mcp/server.py:334`
- **Issue**: `__init__.py` pins `__version__ = "0.1.0"`. `pyproject.toml` declares `version = "0.3.0"`. The MCP `serverInfo` block in `_handle` hardcodes `"version": "0.1.0"`. These are three different version strings. Claude Desktop/Code may report the server version to users as `0.1.0` even after an upgrade to `0.3.0`, causing confusion when diagnosing issues or verifying which version is installed.
- **Fix**: Make version a single source of truth. In `pyproject.toml` use `dynamic = ["version"]` with `setuptools-scm`, or read `importlib.metadata.version("jarvis-trading-friend-mcp")` at runtime in `server.py` and `__init__.py`. At minimum, manually sync all three strings on every release.

### No timeout on stdin reads — process hangs indefinitely if Claude closes stdout without closing stdin
- **File**: `friend_mcp/server.py:382`
- **Issue**: `for line in sys.stdin:` blocks forever until EOF on stdin. In normal operation Claude Desktop/Code closes stdin when it tears down the MCP process, so this resolves cleanly. However, if the host process crashes, hangs, or its pipe is half-closed (stdout closed, stdin left open), the MCP server process remains alive indefinitely consuming resources and holding the token active. This is a low-probability but unrecoverable hung-process scenario.
- **Fix**: Add a `select`/`signal`-based idle timeout, or set `sys.stdin` to non-blocking and handle `BlockingIOError`. Alternatively, document that the MCP relies entirely on the host for lifecycle management and add a note to README that the process is not self-terminating.

### `get_fit_score` comment admits the implementation is a workaround, not the intended design
- **File**: `friend_mcp/server.py:179-183`
- **Issue**: The inline comment explicitly acknowledges "No direct endpoint — the universe scan returns it per candidate. To get a single ticker, use the thesis endpoint and pull `contract.fit_score` from the response. But the cleanest single-ticker call we have is the screener's `directional_fit` via the screener endpoint." This is a known design debt — the function is not backed by the endpoint its name implies. When the upstream API adds a proper `/api/fit-score` endpoint (or the `thesis` approach is deemed better), the bridge code here will need updating. Currently there is no TODO/tracking comment, and the tool description Claude receives does not disclose this limitation.
- **Fix**: Add a `# TODO: replace with dedicated endpoint when available` comment. Consider updating the tool's `description` field to note "score is extracted from the screener response" so Claude's behavior is accurate to the implementation. Track this as a known limitation in CLAUDE.md or a GitHub issue.

## Low

### `JARVIS_TRADING_MAX_CALLS_PER_MIN` parsed with bare `int()` at module load — crashes the process on invalid input
- **File**: `friend_mcp/server.py:68`
- **Issue**: `int(os.environ.get("JARVIS_TRADING_MAX_CALLS_PER_MIN", "30"))` is evaluated at import time. If a user sets this env var to a non-integer value (e.g. `"unlimited"` or `"30/min"`), the server process exits with an unhandled `ValueError` before it can send any JSON-RPC response, and Claude surfaces a generic "MCP server failed to start" error with no actionable message.
- **Fix**: Wrap in a try/except with a fallback and a warning to stderr: `try: RATE_LIMIT_MAX_CALLS = int(...) except ValueError: RATE_LIMIT_MAX_CALLS = 30; print("Warning: invalid JARVIS_TRADING_MAX_CALLS_PER_MIN, using default 30", file=sys.stderr)`.

### `__init__.py` docstring describes 2–3 friends but package is public on PyPI-installable GitHub
- **File**: `friend_mcp/__init__.py:6`
- **Issue**: The module docstring says "For 2-3 trusted friends to access…". README describes the same intent. But the package is installable by anyone via `pip install git+https://github.com/bradylandry/jarvis-trading-mcp.git` — the only guard is the token. This is not a security gap (tokens gate access), but the docstring creates misleading expectations about the audience and may cause confusion when the user count grows or the package is shared more broadly.
- **Fix**: Update the docstring to reflect the actual access model ("token-gated; contact Brady for a token") rather than implying a hard headcount cap that is not enforced in code.
