## Medium

### Unguarded `int()` cast on env var crashes MCP process at import time
- **File**: `friend_mcp/server.py:68`
- **Issue**: `RATE_LIMIT_MAX_CALLS = int(os.environ.get("JARVIS_TRADING_MAX_CALLS_PER_MIN", "30"))` is evaluated at module import time. If the env var is set to a non-integer value (e.g. `"60/min"`, `""`, or a typo), Python raises `ValueError` before `main()` is ever called — the MCP process silently dies and Claude reports a connection error with no useful message.
- **Fix**: Wrap with a try/except and fall back to the default, or validate after parsing:
  ```python
  try:
      RATE_LIMIT_MAX_CALLS = int(os.environ.get("JARVIS_TRADING_MAX_CALLS_PER_MIN", "30"))
  except ValueError:
      RATE_LIMIT_MAX_CALLS = 30
  ```

## Low

### Hardcoded `"version": "0.1.0"` in MCP `initialize` response is stale
- **File**: `friend_mcp/server.py:332`
- **Issue**: The MCP `initialize` response returns `"serverInfo": {"name": "friend_mcp", "version": "0.1.0"}`. The package is at `0.3.0` (`pyproject.toml:6`), and `friend_mcp/__init__.py:15` also declares `__version__ = "0.1.0"`. All three are out of sync; the two source files are two major versions behind.
- **Fix**: Read the version from a single source of truth. The simplest approach is to update `__init__.py` to `__version__ = "0.3.0"` and import it in `server.py`:
  ```python
  from friend_mcp import __version__
  # ...
  "serverInfo": {"name": "friend_mcp", "version": __version__},
  ```
  Or use `importlib.metadata.version("jarvis-trading-friend-mcp")` at runtime so it always matches `pyproject.toml` without any manual sync.
