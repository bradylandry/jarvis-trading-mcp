## Medium

### Version skew: `__init__.py` and `serverInfo` hard-code v0.1.0; package is v0.3.0
- **File**: `friend_mcp/__init__.py:15` and `friend_mcp/server.py:334`
- **Issue**: `__version__ = "0.1.0"` in `__init__.py` and `"version": "0.1.0"` in the MCP `initialize` response are both frozen at the initial release value. `pyproject.toml` is at `0.3.0`. Any client that reads `serverInfo.version` (or calls `from friend_mcp import __version__`) will see a stale value, making it impossible to confirm which release is running without inspecting the file directly.
- **Fix**: Read the canonical version from package metadata in both places. In `__init__.py`: `from importlib.metadata import version; __version__ = version("jarvis-trading-friend-mcp")`. In `server.py` `initialize` handler: import and use `__version__` rather than a string literal.

### Unguarded `int()` at module import time crashes server on invalid env var
- **File**: `friend_mcp/server.py:68`
- **Issue**: `RATE_LIMIT_MAX_CALLS = int(os.environ.get("JARVIS_TRADING_MAX_CALLS_PER_MIN", "30"))` is evaluated at import time. A typo in the env var value (e.g., `"30 "` with a trailing space, `"unlimited"`, or an empty string) raises `ValueError` before `main()` is ever entered. The MCP process dies silently; Claude reports the server as unavailable with no helpful message.
- **Fix**: Wrap in a try/except with a fallback:
  ```python
  try:
      RATE_LIMIT_MAX_CALLS = int(os.environ.get("JARVIS_TRADING_MAX_CALLS_PER_MIN", "30"))
  except ValueError:
      RATE_LIMIT_MAX_CALLS = 30
  ```

### `get_fit_score` tool description promises per-component evidence the endpoint doesn't deliver
- **File**: `friend_mcp/server.py:257–264` (TOOLS list entry)
- **Issue**: The tool description exposed to Claude says "Returns score + per-component evidence (sentiment, fundamentals, Kimi earnings, Kimi-vs-analyst disagreement, insider activity)." The implementation (line 183) calls `/api/screener` — which is the universe screener endpoint, not a dedicated per-ticker fit-score endpoint. The inline comment at line 179 already acknowledges this is a workaround. If the screener response doesn't include those per-component fields for a single ticker, Claude will describe data it didn't receive.
- **Fix**: Update the tool description to accurately describe what `/api/screener?tickers=X` returns, or add a note that per-component breakdown is not available for single-ticker calls. Consider also aligning the function-level docstring at line 175 with the TOOLS entry.
