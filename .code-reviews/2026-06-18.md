## High

### `get_fit_score()` silently drops the `opt_type` parameter
- **File**: `friend_mcp/server.py:183`
- **Issue**: `get_fit_score(ticker, opt_type)` accepts an `opt_type` argument and the tool schema exposes it to Claude, but the implementation calls `/api/screener` with only `{"tickers": ticker.upper()}`. The direction (call/put) is never sent to the API. Every invocation returns the same result regardless of whether the user asked for a bullish or bearish fit. This is a silent correctness bug — no error, just wrong data.
- **Fix**: Forward the parameter: `return _api_get("/api/screener", {"tickers": ticker.upper(), "type": opt_type})`. If the screener endpoint does not accept a `type` parameter, the tool schema and docstring must be updated to remove `opt_type` so Claude stops presenting a non-functional option to users.

### Broad `except Exception` in `_api_get()` leaks internal exception details to Claude
- **File**: `friend_mcp/server.py:148-149`
- **Issue**: The catch-all handler returns `{"error": f"{type(e).__name__}: {e}"}`. For many Python exceptions (e.g., `SSLError`, `ConnectionError`, `JSONDecodeError`) the `str(e)` representation includes internal paths, partial response bodies, or verbose stack context. This string is serialised verbatim into Claude's context. While unlikely to leak credentials, it bypasses the intentional sanitisation applied to non-200 HTTP responses (lines 135-144) and could surface confusing or noisy diagnostic text.
- **Fix**: Either log the full exception to stderr and return a generic message (`"error": "Unexpected error contacting the API. Try again or contact Brady."`), or at minimum strip the exception message and only include the type name. At least extend the explicit `requests` exception handling to cover `requests.ConnectionError` and `requests.exceptions.SSLError` before the generic fallback.

## Medium

### Version string is stale in `__init__.py` and `_handle()`
- **File**: `friend_mcp/__init__.py:15`, `friend_mcp/server.py:332`
- **Issue**: `pyproject.toml` declares version `0.3.0`, but `__init__.py` still says `__version__ = "0.1.0"` and the MCP `serverInfo` in `_handle()` hardcodes `"version": "0.1.0"`. These are three independent sources of truth that are now inconsistent. Tooling that reads `__version__` (e.g., `pip show`, introspection scripts) will report the wrong version, and the MCP handshake will advertise an outdated version to Claude.
- **Fix**: In `__init__.py` change to `__version__ = "0.3.0"`. In `server.py` either import it (`from friend_mcp import __version__`) or use `importlib.metadata.version("jarvis-trading-friend-mcp")` to make it a single source of truth going forward. Consider using `importlib.metadata` so `pyproject.toml` is always canonical.

### `get_fit_score()` comment says "no direct endpoint" but uses a different one without documenting the mapping
- **File**: `friend_mcp/server.py:179-183`
- **Issue**: The function comment says "No direct endpoint — the universe scan returns it per candidate. To get a single ticker, use the thesis endpoint…" but then calls `/api/screener` instead. The comment is outdated and contradicts the implementation. This makes it hard to reason about what data the function actually returns and whether it truly contains `directional_fit` per the docstring promise.
- **Fix**: Update the comment to accurately describe why `/api/screener` is the chosen endpoint and what field in its response corresponds to fit score. If the screener endpoint does not return per-component evidence (sentiment, fundamentals, etc.) as promised in the tool description, the tool description must be corrected.

### No input validation on user-supplied tool arguments before API call
- **File**: `friend_mcp/server.py:163-219`
- **Issue**: Tool functions accept raw arguments from Claude (`ticker`, `strike`, `expiration`, `dte`, `mode`, `opt_type`) and forward them directly to `requests` without any validation. While `requests` URL-encodes params safely (preventing query-string injection), there is no bounds or format checking: e.g. a negative `dte`, a malformed `expiration` like `"YYYY/MM/DD"`, or an empty string `ticker` will all silently produce API calls that return opaque error responses. This complicates debugging and could cause confusing user-facing errors.
- **Fix**: Add lightweight guards at the top of each function (e.g., check that `ticker` is non-empty, `expiration` matches `YYYY-MM-DD`, `dte > 0`). Returning a structured `{"error": "Invalid expiration format — use YYYY-MM-DD"}` is friendlier than propagating a backend 400 whose body is now sanitised away.
