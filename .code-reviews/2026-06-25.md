## High

### `get_fit_score` ignores `opt_type` — wrong endpoint, direction argument dropped
- **File**: `friend_mcp/server.py:183`
- **Issue**: `get_fit_score` is documented as returning a directional fit score for a caller-specified option direction (`call` vs `put`). However the implementation calls `/api/screener?tickers=<TICKER>` and never passes `opt_type`. The comment on line 179-182 acknowledges this is a workaround ("No direct endpoint — use screener") but the workaround fails the contract: the function silently ignores the direction argument. A user asking "Is MSTR a bearish setup?" and one asking "Is MSTR bullish?" will receive identical responses.
- **Fix**: Either (a) find and use the correct endpoint that accepts direction, updating the function signature and schema accordingly; (b) pass `opt_type` as a query param to `/api/screener` if the server supports it; or (c) update the tool description to honestly state that the result is direction-agnostic and remove `opt_type` from the schema to avoid misleading Claude. Do not silently accept a parameter that has no effect.

### `urljoin` path handling — paths without leading `/` silently hit wrong URL
- **File**: `friend_mcp/server.py:127-128`
- **Issue**: `urljoin(API_BASE, path)` behaves unexpectedly when `path` does not start with `/`. For example, `urljoin("https://trading.landrycmd.com", "api/options/scan")` → `"https://trading.landrycmd.com/api/options/scan"` (happens to work here because `API_BASE` has no path component). But if `JARVIS_TRADING_API` is ever set to a value with a path component (e.g. `https://trading.landrycmd.com/v2`), `urljoin` will replace the path entirely for relative `path` values, while an absolute-path `path` (starting with `/`) would strip the `/v2` prefix. All current `path` values do start with `/`, so this is not currently broken — but it is a latent footgun. The comment on lines 118-126 credits `urljoin` as "strictly safer" without noting this subtle behavior.
- **Fix**: Assert or enforce that `path` always starts with `/` at the top of `_api_get`, or use `API_BASE.rstrip("/") + path` with an explicit assertion that `path` starts with `/`. This makes the contract explicit and immune to `urljoin` path-replacement semantics.

## Medium

### Version constant in `__init__.py` is stale and diverges from `pyproject.toml`
- **File**: `friend_mcp/__init__.py:15`
- **Issue**: `__version__ = "0.1.0"` but `pyproject.toml` declares `version = "0.3.0"`. The package has shipped two subsequent releases (0.1.1, 0.3.0 per pyproject history) without updating the in-code constant. Any code or tooling reading `friend_mcp.__version__` — including the `serverInfo.version` field returned to Claude in the MCP `initialize` handshake — reports `0.1.0` instead of `0.3.0`.
- **Fix**: Either (a) set `__version__ = "0.3.0"` and keep them in sync manually on each release, or (b) use `importlib.metadata.version("jarvis-trading-friend-mcp")` to derive the version from the installed package metadata, eliminating the duplication. Option (b) is more robust.

### `serverInfo.version` in `initialize` response is hardcoded and also stale
- **File**: `friend_mcp/server.py:334`
- **Issue**: The `initialize` response hardcodes `"version": "0.1.0"` independently of both `pyproject.toml` and `__init__.py`. Three places now track the version with three different values or stale values. Claude Desktop surfaces this version in its MCP server panel; users and Brady cannot tell which version a friend has installed.
- **Fix**: Derive from `friend_mcp.__version__` (once that is fixed per the Medium finding above), or use `importlib.metadata.version(...)` directly here.

### `docs/entra-oauth-migration.md` contains the operator's personal email address in a public repo
- **File**: `docs/entra-oauth-migration.md:19`
- **Issue**: The file contains `bjlandry@gmail.com` in plaintext in a public GitHub repository. While this is not a secret/credential, it is PII that could attract spam, social-engineering attempts, or be harvested by scrapers. The email is the bootstrap identity for a future Microsoft Entra tenant.
- **Fix**: Replace with a placeholder (e.g., `<your-gmail>`) or remove the parenthetical. This is a docs-only change.

## Low

### `main()` silently discards JSON parse errors — no stderr logging
- **File**: `friend_mcp/server.py:388-389`
- **Issue**: Malformed lines from stdin (which could include legitimate MCP messages that got corrupted) are silently `continue`d with no stderr logging. This makes debugging initialization failures or misconfigured clients very difficult.
- **Fix**: Add `sys.stderr.write(f"[friend_mcp] JSON parse error on line: {line!r}\n")` before `continue` so the process at least emits a diagnostic to the process log visible in Claude Desktop's MCP debug view.

### `RATE_LIMIT_MAX_CALLS_PER_MIN` env var name does not match the window constant
- **File**: `friend_mcp/server.py:68-69`
- **Issue**: The env var is `JARVIS_TRADING_MAX_CALLS_PER_MIN` but the window variable is `RATE_LIMIT_WINDOW_S = 60`. The name implies a per-minute cap, which is correct today. But if the window is ever changed, the env var name becomes misleading. Minor naming clarity issue.
- **Fix**: Consider renaming the constant to `RATE_LIMIT_MAX_CALLS` and the env var to `JARVIS_TRADING_MAX_CALLS` with an updated description, or keep the current name but add a comment that the `_PER_MIN` suffix assumes `RATE_LIMIT_WINDOW_S == 60`.

### `__init__.py` `__version__` not re-exported or used by any consumer
- **File**: `friend_mcp/__init__.py:15`
- **Issue**: `__version__` is defined but never imported or referenced by `server.py`, which independently hardcodes its own version string. The constant is dead code in its current state (separate from the staleness issue noted under Medium).
- **Fix**: Once version derivation is consolidated (see Medium findings), wire the single source of truth into `server.py`'s `initialize` response.
