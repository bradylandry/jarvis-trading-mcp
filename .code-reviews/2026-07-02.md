## High

### `get_fit_score` silently ignores `opt_type` — always returns undifferentiated result
- **File**: `friend_mcp/server.py:183`
- **Issue**: `get_fit_score(ticker, opt_type)` accepts an `opt_type` parameter and the tool schema advertises directional results ("Directional fit score (0-10) for ticker + direction"), but the implementation calls `/api/screener` with only `{"tickers": ticker.upper()}` — `opt_type` is never forwarded. Every call returns the same payload regardless of whether the user is evaluating a `call` or `put`. A user who asks "is MSTR a bearish put setup?" gets the same undifferentiated screener response as "is MSTR a bullish call setup?", silently.
- **Fix**: Pass `opt_type` to the API: `_api_get("/api/screener", {"tickers": ticker.upper(), "type": opt_type})`. If the upstream `/api/screener` endpoint does not accept a direction filter, the tool description must be corrected and the parameter removed to avoid misleading callers. Either way the silent no-op must be resolved.

### `RATE_LIMIT_MAX_CALLS_PER_MIN` env var parsed at module import — invalid value crashes MCP silently
- **File**: `friend_mcp/server.py:68`
- **Issue**: `int(os.environ.get("JARVIS_TRADING_MAX_CALLS_PER_MIN", "30"))` is evaluated at module import time. If the env var is set to a non-integer value (e.g. `"120calls"`, a copy-paste mistake), `int()` raises `ValueError` and the entire MCP process fails to start. The Claude host receives no response and the user sees a generic "MCP failed to start" error with no useful message.
- **Fix**: Wrap the parse in a try/except at module level and fall back to the default with a stderr warning, or defer the parse to `main()` where a clean error can be emitted before exiting:
  ```python
  try:
      RATE_LIMIT_MAX_CALLS = int(os.environ.get("JARVIS_TRADING_MAX_CALLS_PER_MIN", "30"))
  except ValueError:
      sys.stderr.write("JARVIS_TRADING_MAX_CALLS_PER_MIN must be an integer; defaulting to 30\n")
      RATE_LIMIT_MAX_CALLS = 30
  ```

### Version string inconsistent across three locations
- **File**: `friend_mcp/__init__.py:15`, `friend_mcp/server.py:334`, `pyproject.toml:6`
- **Issue**: `pyproject.toml` declares `version = "0.3.0"`. `__init__.py` declares `__version__ = "0.1.0"`. The MCP `initialize` response (in `_handle`) hardcodes `"version": "0.1.0"`. Three sources of truth, all different. Any client or diagnostic tool that reads the MCP's `serverInfo.version` will see `0.1.0` even on the current release.
- **Fix**: Read a single authoritative version from `importlib.metadata`:
  ```python
  from importlib.metadata import version as _pkg_version
  __version__ = _pkg_version("jarvis-trading-friend-mcp")
  ```
  Use `__version__` in `serverInfo` and remove the hardcoded string. This requires `importlib.metadata` (stdlib ≥ 3.8, satisfying the `>=3.9` constraint).

### Bare `except Exception` in `_api_get` leaks internal error strings into Claude context
- **File**: `friend_mcp/server.py:148`
- **Issue**: The sanitized-error approach (introduced 2026-05-08) correctly suppresses the HTTP response body on non-200 responses, but the generic `except Exception as e` on line 148 passes `str(e)` directly into the error dict that reaches Claude. TLS errors and connection failures can embed hostnames, IP addresses, certificate subjects, or internal socket details in their string representation — bypassing the intended sanitization.
- **Fix**: Add targeted catches before the bare `except` for the most common non-HTTP failures:
  ```python
  except requests.exceptions.SSLError:
      return {"error": "TLS/SSL error connecting to trading API. Contact Brady."}
  except requests.exceptions.ConnectionError:
      return {"error": "Could not connect to trading API. Check your network."}
  except Exception:
      return {"error": "Unexpected error calling trading API. Contact Brady."}
  ```

### No input validation on `expiration` date format before forwarding to API
- **File**: `friend_mcp/server.py:163`
- **Issue**: `get_thesis` accepts any string as `expiration` with no format check. The docstring and tool schema say `YYYY-MM-DD`, but a value like `"May 29"` or `"2026/05/29"` is forwarded to the upstream API. The API will likely return a 400 or 422, which surfaces back as `"API returned status 4xx"` — a confusing error for the user. A malformed ticker (e.g. empty string) has the same problem.
- **Fix**: Add lightweight pre-flight validation in the tool functions:
  ```python
  import re
  if not re.fullmatch(r"\d{4}-\d{2}-\d{2}", expiration):
      return {"error": "expiration must be in YYYY-MM-DD format (e.g. '2026-05-29')."}
  if not ticker or not ticker.isalpha():
      return {"error": "ticker must be a non-empty alphabetic symbol (e.g. 'NVDA')."}
  ```
